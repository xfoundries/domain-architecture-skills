Rules:
- This is a package-role template, not a mandatory Maven module layout.
- domain contains aggregates, entities, value objects, domain events, domain services, aggregate repository contracts, and optional narrow secondary ports consumed by domain policies/services.
- application contains use cases, application services, primary ports, workflow/read secondary ports, commands, and query models, organized by business capability before architectural role.
- Global application/port/in and application/port/out packages remain valid for small projects. JFoundry recognizes direction packages at any depth; the capability-first shape is a recommendation for non-trivial applications, not a Hexagonal Architecture mandate.
- Primary ports do not depend on models owned by port/out, and secondary ports do not depend on models owned by port/in. Put shared query/view models in a neutral application capability package such as query/view.
- Port and Adapter are Hexagonal roles, not the first source of domain names. Use ubiquitous language first, and use Port/Adapter suffixes only when they improve clarity.
- if CQRS is selected, place command/query packages below the business capability rather than mixing top-level command/query/port/service organization axes.
- for a command exposed to a primary adapter, place the business-named `*UseCase` `@PrimaryPort` in command/port/in and let its command handler implement it. Do not add a fixed forwarding `*CommandDispatcher`; reserve dispatchers for genuine runtime command routing.
- Select one adapter direction vocabulary for the entire Hexagonal project: this template uses adapter/in and adapter/out. adapter/primary and adapter/secondary is an equivalent alternative, but do not mix the pairs outside an explicitly bounded migration. Add the matching JFoundry ArchUnit rule from the Java template.
- adapter/in contains primary adapters that call primary ports or application services. Keep HTTP/API Request/Response DTOs adapter-local under web feature request/response packages; place message listeners and schedulers in their sibling messaging and scheduler packages.
- adapter/out contains secondary adapters that implement secondary ports from domain or application modules. Group by technical shape first and business feature or external system second: persistence/aggregate-or-feature for aggregate lifecycle storage, lookup/feature for command-side fact reads, query/feature for caller-facing read-only views, and projection/feature for event/state-change-driven read-model materialization, plus client/external-system, messaging/topic-or-system, file/feature, and cache/feature. Prefer query over readmodel unless the project explicitly uses read-model terminology. Do not create every category by default. It contains no domain logic and no global runtime framework configuration when a runtime assembly package/module exists.
- boot contains the runtime entry point, selected runtime framework starters, runtime configuration, component scanning, and bean wiring. The package name may differ outside Spring Boot projects.
